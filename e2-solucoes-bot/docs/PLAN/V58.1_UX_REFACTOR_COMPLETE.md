```javascript
// Helper function (add to State Machine Logic)
function formatPhoneDisplay(phone) {
    // Format: (62) 98765-4321
    if (phone.length === 11) {
        return `(${phone.substr(0,2)}) ${phone.substr(2,5)}-${phone.substr(7,4)}`;
    } else if (phone.length === 10) {
        return `(${phone.substr(0,2)}) ${phone.substr(2,4)}-${phone.substr(6,4)}`;
    }
    return phone;
}
```

## ⚠️ Error Handling Patterns (Gap #7 ✅)

**Problem**: V58 plan didn't specify error handling for new states.

**Solution**: Document error handling pattern following V57.2 approach.

### Error Handling Strategy

**For all V58.1 new states**, apply consistent error handling:

1. **Invalid Input Detection**: Check validation in each case
2. **Error Message**: Show "❌ *Opção inválida*" + repeat original prompt
3. **Stay in Current State**: `nextStage = currentStage`
4. **Increment Error Counter**: Track repeated errors
5. **Max Errors Threshold**: After 3 errors, offer menu or human handoff

**Example** (collect_phone_whatsapp_confirmation):

```javascript
case 'collect_phone_whatsapp_confirmation':
    if (message === '1' || message === '2') {
        // Valid input - process normally
    } else {
        // Gap #7 - Error handling ✅
        console.log('V58.1 ERROR: Invalid input for phone confirmation');

        const errorCount = (currentData.errorCount || 0) + 1;

        if (errorCount >= 3) {
            // Max errors reached - offer help
            responseText = `❌ *Desculpe, tivemos dificuldade*\n\nVocê gostaria de:\n\n1️⃣ - Voltar ao menu principal\n2️⃣ - Falar com um atendente\n\n💡 _Responda 1 ou 2_`;
            nextStage = 'error_recovery';
        } else {
            // Standard error - repeat question
            responseText = `❌ *Opção inválida*\n\n${templates.collect_phone_whatsapp_confirmation.replace('{{phone}}', currentData.phone_number)}`;
            nextStage = currentStage;
        }

        updateData.errorCount = errorCount;
    }
    break;
```

**Error Recovery State** (NEW - optional enhancement):

```javascript
case 'error_recovery':
    if (message === '1') {
        // Return to menu

        // Handoff to human
        responseText = '👤 *Transferindo para atendente...*\n\nEm breve um de nossos especialistas entrará em contato!';
        nextStage = 'handoff_comercial';
        updateData.errorCount = 0;
    }
    break;
```

## 🎓 Lessons Learned

### From Gap Analysis

1. **State Name Mapping Critical**: Database vs code state names MUST be mapped
2. **Validator Completeness**: Every state needs explicit validator mapping
3. **State Flow Completeness**: Design ALL states in flow, not just happy path
4. **Template Completeness**: Every state needs template, including error paths
5. **Architecture Documentation**: Preserve patterns must be explicit (not assumed)
6. **Field Mapping**: Don't assume fields will "just work" - verify database mappings
7. **Service Selection**: Number vs STRING mismatch - always map enums
8. **Error Handling**: Must be specified for every state, not just "normal" ones

### For Future Refactoring

1. **Gap Analysis First**: Analyze before implementing (save time, avoid bugs)
2. **Incremental Changes**: Add complexity gradually, test thoroughly
3. **Preserve Architecture**: Critical patterns (Merge Append, Process Data) are delicate
4. **Complete Flows**: Design entire user journey, including error recovery
5. **Database Verification**: Always verify field population after implementation
6. **Compatibility Testing**: Test with existing data, not just new conversations
